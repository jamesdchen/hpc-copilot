## Findings (severity-ordered)

**F1 — `daemon_client.py` "stdlib-only" is unimplementable without forking two definitions (one-definition, CONFIRMED).** The client must compute the fingerprint-embedding address (needs `_build_info.full_version()`) and, per §2, hello carries "the client's `HEALABLE_TRANSPORT_ENV_VARS` values" (needs `env_flags.py:44`). A stdlib-only module can only get these by COPYING them — exactly the mirror-twin class `lint_mirror_ledger.py` exists for. Fixes: (a) let the client import `_build_info` (the one definition) and replace "stdlib-only" with an enforced import-weight budget (contract test capping daemon_client's transitive module set / `-X importtime`); (b) delete the client-partitioned `transport_env` hello field — the client already sends ALL `HPC_*` verbatim in `hpc_env`; the DAEMON, which owns `env_flags`, partitions transport vars using the one definition. No list crosses the pipe pre-partitioned.

**F2 — Rung 1 presumes hook bodies are dispatchable verbs; most are not (one-definition, CONFIRMED).** Hooks today are in-process package code (`utterance_capture.py:85-86` imports `env_actor`/`append_utterance` directly), not CLI dispatches. "Hook head forwards `x-hpc/cli` with the hook's payload" requires a CLI verb per hook body. Where none exists, rung 1 either creates one (a NEW agent-reachable affordance unless registered `agent_facing=False`) or the hook keeps a parallel in-process copy (mirror twin). Required spec addition: per hook, NAME the verb that IS the hook body (hook module becomes a thin client of it; `relay_audit_stop`→`verify-relay` per row 250 is the model), register each `agent_facing=False`, and pin hook-path/verb-path output parity. Without this, rung 1 forks guard logic — the highest-consequence silent divergence in the design.

**F3 — ContextVar `call_env()` does not propagate to verb-spawned threads (per-call context, CONFIRMED).** A fresh `threading.Thread` starts with an empty context → `call_env()` returns the default `os.environ` — the DAEMON's env, not the call's. §4 asserts this is "correct" for engine/sweeper/heartbeat threads (true), but a Lane-A verb body spawning a helper thread that reads any `HPC_*` silently reads the wrong env with no refusal. The eligibility criteria must gain a mechanized leg: no `threading.Thread`/`ThreadPoolExecutor` in Lane-A-eligible verb bodies (allowlist with justification), or the lane snapshots `hpc_env` into the context of every thread it spawns. Fire test: planted Lane-A verb spawning an env-reading thread → red.

**F4 — Hardening delta 1 (fsync_required) changes the SHARED seam, so the permanent one-shot fallback is no longer byte-identical (journal authority / fallback doctrine, CONFIRMED).** Splitting `append_jsonl_line` callers makes a suppressed-fsync `OSError` surface as ok:false on the one-shot CLI too (today: ok:true). It is the right correctness fix for both paths, but it contradicts the letter of "the CLI one-shot fallback remains byte-identical and permanent." Required: an explicit RULING line amending the byte-identity claim ("byte-identical modulo the shared durability fix, ruled <date>"), row 313 amended in the same wave, and `test_write_marker_swallows_seam_oserror_and_harvest_continues` pinned to stay green (harvest marker keeps suppression).

**F5 — Editable-install "stat of the package root's key files" is an undeclared judgment heuristic (zero-judgment axis).** Which files are "key" is a call that silently misses a `git pull` touching others — incident 2 half-closed. Mechanize or refuse: v1 recommendation is editable/dirty installs are daemon-INELIGIBLE (client falls inline — the memo §0.10 dirty-checkout-always-walks precedent, devs pay 1.3s), or a declared whole-tree newest-mtime snapshot. Either way: enumerated, fire-tested, never "key files."

**F6 — Never-unpickle and authkey are prose, not pins (security mechanization).** `Connection.recv()` unpickles; one future `send_bytes`→`send` edit reopens the RCE hole. Required: AST/lint pin that daemon_server + daemon_client use only `recv_bytes`/`send_bytes`; fire test that a pickle-bytes payload gets a JSON-parse refusal; fire test that a wrong authkey is refused (prove the HMAC challenge fires); POSIX 0600/0700 discovery-file pin (Linux CI gates; win32 disclosure stands as written — honest).

**F7 — "The in-process runner stays THE runner" needs a route-through pin, not prose (one-definition).** Per map conventions (the `test_layers_share_one_drift_predicate` / `inspect.getsource` pattern): mechanized assertion that the `x-hpc/cli` handler routes through `_in_process_cli_runner` (Lane A) / `run_capture_bounded` + `python -m` (Lane B), so a "daemon RPC dispatcher v2" that re-implements verb dispatch turns a test red.

**F8 — Daemon-never-elicits needs a structural pin so the guard can fire pre-runtime (elicitation axis).** §3.5 is correct (verified: `_elicitation_applies` :1283 gates on per-session declared capability, which the pipe surface never declares; `_elicit_then_retry` :1312 is refusal-internal, model never sees it — the shim flow preserves that shape). Add: import/grep pin that `daemon_server.py` never imports the elicitation machinery, plus the §8 locality test. The re-pointed E4 tests must include pipe-originated `append-decision` authorship refusal byte-identical to today's CLI refusal, and retry-once bound in the shim.

**F9 — `state/journal.py:169` assert-on-daemon guard: mechanism and fire path unnamed.** How state code knows it is daemon-hosted must be a named mechanism (a lane-set contextvar marker, not a global flag verbs could miss), with the fire test named in §8: daemon dispatch reaching the `experiment_dir is None` cwd-fallback refuses loudly, never journals under the daemon's `~`.

**F10 — Supervisor wake fan-out must carry the wake-is-a-hint law (memo §0.4, doctrine row 11).** "Notifying a waiting client when a lease's terminal lands" must never substitute for the client's own settle read of the durable lease/terminal — restate + pin. Also unspecified: how a client "waits" without occupying Lane A (a subscription outside the lane is a new concurrency surface — declare it).

**F11 — `pid_alive` in the client head costs psutil (tension, decide loudly).** The one definition (`infra/proc.pid_alive`) routes through psutil — heavy for a rung-1 hook head whose import weight "IS the product." Either accept psutil in the client (and let the budget test price it) or declare a bounded client-side probe as a NAMED second definition with a mirror-ledger entry. Do not let this get decided silently at build time.

**F12 — WEDGED-shoot justification is factually wrong as written ("verify a guard" honesty rule).** "A wedged daemon holding no locks … is safe to shoot" — the in-flight verb body may hold the journal flock at `os._exit` time. The mechanism IS safe, for three citable reasons: the OS releases the flock on death (`io.py:377` backend), appends are whole-line-atomic with a backward-scanning reader (row 313), and the client treats no-reply as UNKNOWN + re-drives (§5). Fix the prose to cite those, not the false no-locks claim.
