LIFECYCLE/WINDOWS PREMORTEM — WS-DAEMON v2. Findings ranked by severity; code claims verified against main @ 57b31f9c and the venv's stdlib.

## HIGH

**H1. The "hard client deadline ≈ 250ms connect+hello" is not implementable with `multiprocessing.connection.Client` as written — a wedged daemon stalls every surface up to 20s, not 250ms.**
Verified in this venv's stdlib: `Client()` takes no timeout; on win32 `PipeClient` loops `WaitNamedPipe(address, 1000)` retrying on `ERROR_PIPE_BUSY`/`ERROR_SEM_TIMEOUT` against `CONNECTION_TIMEOUT = 20.` seconds. The exact wedge scenario the design worries about (daemon alive, Lane A occupied, accept thread stalled or listener instance exhausted) presents as PIPE_BUSY → every hook and CLI call burns up to 20s *inside stdlib*, uninterruptible from Python. `recv` is also unbounded (no timeout parameter), so a daemon that accepts + never replies to hello wedges the client at recv.
This re-creates incident 3's inversion failure: the daemon's wedge becomes every surface's wedge, ×3 hooks per turn.
Mitigation (must be in the build spec, not left to discovery): (a) connect on an abandonable helper thread — caller waits `deadline_ms`, abandons; the leaked thread dies with the short-lived client process (fine for CLI/hooks; the rung-3 shim must cap/reuse); or drive `_winapi.CreateFile`/`WaitNamedPipe(address, 250)` directly in `daemon_client.py` (it's already stdlib-private-adjacent; wrap once). (b) After connect, never bare `recv()` — always `conn.poll(remaining_deadline)` first, for hello and for the reply. (c) A fire test: daemon with a planted stuck accept loop → hook-shaped client returns inline within its deadline on Windows.

**H2. Discovery-file pid reuse (the design's step (b) uses bare `pid_alive`) — the repo already learned this lesson and the design doesn't reuse it.**
`_kernel/lifecycle/detached.py:320-350` (F43) pairs pid with `psutil.Process(pid).create_time()` precisely because a reused pid reads alive forever. The daemon discovery check as specified: daemon dies, Windows recycles the pid (aggressively — small space, multiples of 4), some unrelated long-lived process now owns it → `pid_alive(pid)` true → client proceeds to connect → connect fails → per the design's own rule "never delete on a mere connect failure," the file is never cleaned. Outcome depends on an ambiguity the design leaves open: if connect-failure-with-live-pid does not trigger activation, the daemon silently never runs again (permanent inline degradation, invisible because fallback is byte-identical); if it does trigger activation, fine but then the "mid-boot" protection rationale is dead code. 
Mitigation: write `create_time` into the discovery JSON at ready; staleness check = `pid_alive(pid) and _create_times_match(recorded, current)` — reuse `_process_create_time`/`_create_times_match` (one-definition: they're the existing pid-identity probe). Then a connect failure with a *verified-identity* live pid is genuinely "mid-boot, retry later," and everything else is stale→clean→respawn. Also specify: `daemon-status` must apply the same identity check, or it reports a squatter pid as a running daemon.

**H3. Boot-crash loop has no breaker — a daemon that fails to boot converts every CLI call into a +15s tax.**
Activation: spawn lock → spawn → bounded wait ≤15s for the discovery file. If `daemon-serve` reliably dies at boot (the exact stale-wheel/import-error class this program exists to fix; also Controlled Folder Access denying the `~/.hpc-agent` discovery write, Defender quarantining a fresh wheel, a pipe-name squatter making `Listener()` raise), every subsequent call re-wins the spawn lock, re-spawns, re-waits 15s, falls inline. Hooks are protected (fire-and-forget) but interactive CLI verbs get *slower than no daemon at all* — the ladder's rung 2 becomes a regression precisely when the install is broken.
Mitigation: a `spawn-failed-<fingerprint12>.json` negative-cache (atomic_write_json: timestamp, exit code, stderr tail) written by the activation loser-path; subsequent activations skip spawn for a backoff window (e.g. 5 min, env-overridable) and fall inline immediately; `doctor` surfaces it with the stderr tail. Also: capture the spawned daemon's stderr to `<daemon_dir>/boot-<fingerprint12>.err` — a DETACHED_PROCESS child has no console, so an uncaught boot exception is otherwise *unobservable* (see H4).

**H4. The daemon process has no valid std handles (DETACHED_PROCESS) — any stray write to stdout/stderr raises `OSError` and can kill threads silently.**
The design spawns the daemon "with the same platform detach flags `_spawn_detached` uses" (`detached.py:216-223`: DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP | CREATE_BREAKAWAY_FROM_JOB). Detached *workers* get their std streams redirected to `log_path` by the launcher; the design never states this for the daemon. With DETACHED_PROCESS and no redirection, `sys.stdout/stderr` are invalid handles: the first heartbeat-style print, logging-to-stderr fallback, or traceback print in the accept thread raises — the stdin-false-EOF class (incident 1) reborn as stdout-false-EBADF. This is the same seam class that killed the server for six days.
Mitigation: the activation spawner MUST pass `stdout=stderr=<daemon log fd>`, `stdin=DEVNULL`; the daemon's first act re-points `sys.stdout/sys.stderr` at its rolling log and installs a `threading.excepthook` that logs + marks unhealthy. Add to §8 a fire test: daemon runs a verb that prints to real stderr → daemon survives and the bytes land in the log.

**H5. Sleep/resume: Windows `time.monotonic` (QPC) ticks *through* suspend — the watchdog and the ssh pool both misbehave after a laptop sleep, and the daemon makes this class far more frequent than mcp-serve did.**
Two concrete failures: (a) laptop sleeps mid-Lane-A call → on resume, in-flight wall time includes the sleep → watchdog reads ceiling×1.5 exceeded → healthy daemon marks WEDGED and shoots itself (self-healing but noisy, and it takes the warm pool with it every single lid-close); (b) the asyncssh pool's TCP connections are silently dead after resume — `IDLE_CLOSE_SEC=120` only helps if the sweeper's clock advanced AND the connection idled; a resume followed immediately by a warm Lane-A cluster read hits a black-holed TCP send and eats the full 20s Lane-A ceiling, head-of-line-blocking the turn's other hooks. A long-lived daemon crosses sleep boundaries daily; mcp-serve sessions mostly didn't.
Mitigation: a 1s-tick suspend detector thread (gap > 5s between ticks = resumed): on resume, (i) reset/forgive the watchdog's in-flight timer once, (ii) invalidate the ssh pool (`shutdown_all` or mark-all-stale so next use redials), (iii) log a lifecycle event. Idle-exit should key off "monotonic since last completed RPC" and is fine either way (exiting early after a sleep is harmless). Add to `daemon-status`: `resumes_detected`.

## MEDIUM

**M1. Drain's "30s hard deadline" leaves Lane-B children unspecified — an orphaned mutating child with an UNKNOWN outcome.**
At wheel-swap drain or WEDGED `os._exit`, an in-flight Lane-B child (`python -m hpc_agent <argv>` doing SSH) can outlive the daemon: reply lost, client re-drives per §5, while the orphan is still executing the same verb — a concurrent duplicate the one-shot world never produces *simultaneously* (today's CLI child dies with its console; the daemon's child is detached from the client entirely). Journal flocks serialize the writes, but two live executions of a non-idempotent-in-flight verb (resolve-resources is safe; anything that stages or submits is not) race semantically.
Mitigation: at drain hard-deadline and at WEDGED exit, tree-kill Lane-B children via the same `run_capture_bounded` kill path before exiting (it already owns taskkill /T), then reply `daemon_stale` where a reply is still possible. Pin with a fire test: drain with a planted slow Lane-B child → child's tree is dead before the daemon's pid is.

**M2. Idle-exit / logout durability claim — VERIFIED, with two named residuals.**
Checked: leases + terminals + journal live under the experiment dir via `append_jsonl_line`/`atomic_locked_update` (io.py:257-374, fsync'd, flocked); detached workers are separate processes with their own engines; the supervisor's watch table is reconstructable from a `_detached/` scan; `_guard_single_lease` self-heals dead pids with host+create_time guards (detached.py:353+). A daemon killed by logout while cluster jobs run loses nothing durable. Residuals to state in the doc: (a) Windows **logoff kills detached workers too** (DETACHED_PROCESS survives console-close and job-close, not session logoff) — that's today's contract, unchanged, but the doc's "everything durable survives" section should say the *workers* die and the *state* survives, and recovery = next session's dead-pid lease reclaim + idempotent harvest re-run; (b) a client blocked on the daemon's wake fan-out at logout gets nothing — fine (client dies too), but a client waiting on wake while the daemon *idle-exits* must be impossible: the "zero open connections" gate covers it only if wake-waiters hold their connection open — make that a stated invariant + test.

**M3. Editable-install staleness check is porous as specified.**
"A stat of the package root's key files" — if the pulled fix touches a file not on the key list, the daemon serves stale code indefinitely with a *passing* identity check, which is worse than no check (it certifies freshness it doesn't have). Recursive newest-mtime per RPC is too slow.
Mitigation: for editable installs, stat exactly two sentinels — `.git/HEAD` and the repo's `.git/index` mtime (any pull/checkout/commit touches them) — plus `_build_info` fingerprint; OR declare editable installs daemon-ineligible (activation refuses, always inline) and record it. Either is honest; the key-file list is not.

**M4. Spawn lock is global but daemons are per-fingerprint — a wheel-swap window serializes and taxes the wrong clients.**
During the exact window two fingerprints legitimately coexist (§1), a slow boot of daemon-A holds `spawn.lock` while a client of fingerprint-B fails the non-blocking acquire and then "polls the same file" — but B's discovery file (`daemon-<fpB>.json`) will never appear; B waits its full bound and falls inline every call until A's boot completes.
Mitigation: `spawn-<fingerprint12>.lock`. One line in the design, eliminates the cross-version coupling.

**M5. Two-racer activation: the instance flock must be the daemon's *first* statement, before package import.**
Boot is seconds (registry load, WS-BAKE pending). If `daemon-serve` acquires the instance lock only after warm-up, two spawned instances both burn full boots (memory + Defender scan amplification ×2) before one loses. Also the loser's "read discovery file and exit 0" can race a winner that hasn't written it yet — loser must poll-with-deadline, not read-once.
Mitigation: a thin bootstrap in `cli/daemon.py`: acquire non-blocking instance flock → only then import the heavy tree; loser polls the discovery file with the same bounded wait the client uses. Fire test (§8 already has the two-racer test — extend it to assert exactly one *full boot*, e.g. via a boot-marker count).

**M6. Torn-final-line is a third crash state §5 doesn't cover.**
WEDGED `os._exit` (or power loss) can land mid-`fh.write(line)` inside the append seam — the flock dies with the process (OS-released, verified io.py:436), leaving a *partial* trailing line. §5's fire tests cover "no record" and "record whole," not "record torn." Journal readers mostly tolerate decode errors (`decision_journal.py:285` et al.) but the behavior isn't pinned as a contract for the trailing-line case, and a subsequent append lands *after* the torn fragment on the same line unless the appender is newline-aware.
Mitigation: extend the seam: on open-for-append, if the file's last byte exists and isn't `\n`, prepend `\n` before the new record (self-healing, append-only preserved); pin reader skip-torn-tail behavior with a test; add kill-mid-write to the §5 fire battery.

## LOW

**L1. Pipe-name squatting is a local-DoS, not a spoof — but boot must fail loudly.** Named-pipe namespace is machine-global; another local user can pre-create the name. multiprocessing's mutual HMAC challenge (authkey both directions) blocks impersonation, and `FILE_FLAG_FIRST_PIPE_INSTANCE` makes the daemon's `Listener()` raise on a squatter. Ensure that raise routes into the H3 negative-cache with the winerror named, and `doctor` explains it.

**L2. `<user>` in the pipe name needs one definition.** `getpass.getuser()` (env-derived: USERNAME) vs `os.getlogin()` can disagree in case or under scheduled tasks — two surfaces deriving differently get two daemons and double-book the per-host ssh slots §1 exists to prevent. Define one seam (casefolded `getpass.getuser()`), use it for pipe name, discovery filename, and daemon dir. Also cap/hash: usernames with spaces/unicode are legal in pipe names but POSIX `sock-` paths hit the AF_UNIX ~104-byte limit under a deep `HPC_DAEMON_DIR` — hash user into the same 12-hex discipline as the fingerprint.

**L3. `~/.hpc-agent/daemon` on a synced/redirected profile.** OneDrive "folder backup" doesn't cover `%USERPROFILE%` root by default, but roaming profiles and `HPC_DAEMON_DIR` pointed at a synced dir would put the *authkey* in cloud sync and put flocks on placeholder files. `doctor` check: daemon dir is local + not reparse-pointed; docs: never point `HPC_DAEMON_DIR` at a synced path.

**L4. Old-daemon lingering after swap is bounded and acceptable — state it.** If no client ever dials the old fingerprint again and the new daemon never boots (flag off), the old daemon sits until idle-exit (≤2h) — but its ssh pool self-drains via `IDLE_CLOSE_SEC=120`, so the cluster-facing cost is ~2 min, and the §6 boot-scan shutdown covers the common case. Add `started_at`+fingerprint to `doctor`'s daemon enumeration so a human can see the lingerer.

**L5. Wheel swap under a live *detached worker* (not the daemon) can frankenstein lazy imports.** Workers pin code identity at spawn via `sys.executable -m hpc_agent` (detached.py:253-271) but import lazily; a reinstall mid-S3-watch swaps module files under a running worker. Not daemon-caused, but `daemon-stop` in the release sequence will *feel* like "safe to install now" — the release step should also print live leases from the `_detached/` scan (the supervisor's table makes this cheap) as a "workers still running old code" disclosure, never a block.

**L6. Daemon job-object membership is environment-dependent — the daemon may silently die with the Claude session.** `_popen_detached`'s documented Win11 behavior (detached.py:200-214): breakaway denied can *silently succeed while keeping the child in the job* → a daemon spawned by a hook dies at session close. Harmless (on-demand respawn, durable state on disk per M2) but telemetry should distinguish "daemon died with session" from crashes: log job-membership best-effort at boot (`psutil`/`IsProcessInJob`) into the lifecycle log so soak analysis doesn't chase phantom crashes.

Cross-cutting verification note: the design's claim that the instance flock has "no stale-lock hazard" is correct (filelock → msvcrt byte-range/fcntl, OS-released on death, io.py:377-446), and the claim that `atomic_write_json` is the right discovery-file writer is correct (io.py:110-174, fsync + parent-dir fsync). The two load-bearing gaps against real code are H1 (stdlib Client has no bounded connect — 20s built-in retry constant confirmed in this venv) and H2 (bare `pid_alive` where the codebase's own F43 machinery already carries the pid+create_time answer).
