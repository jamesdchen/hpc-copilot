# WS-DAEMON v2 — Warm-Daemon Architecture (full engineering design)

**Status**: ARCHITECT design, 2026-07-16, baseline main @ 57b31f9c. Extends (never contradicts) ARCHITECT-MEMO §0.8 + unit-specs 3.3 settled calls: fsync-before-reply (D3), full-fingerprint handshake, `python -m` launch, pipe name embeds user+fingerprint, dist-signature per RPC, `env_actor` from the call's env, hard fallback = guard-integrity law. Where the memo declared "concurrency is explicitly out of this program (request-context accessor is v2)" — this document IS that v2, per the user ruling "exercise won't fix them, engineering will." Ladder: opt-in `HPC_CLI_DAEMON=1`, default flip only after a full proving run. The one-shot CLI path stays byte-identical and permanent.

**Incident evidence → design obligation map** (all seven tonight's inputs are load-bearing below): (1) stdin false-EOF → §3 lane isolation + §8 soak; (2) version skew / stale servers → §1 fingerprint-in-address + §2 per-RPC identity + §6 drain-restart; (3) serial dispatch → §3 two lanes + task supervisor; (4) process-global stream redirect → §3 "serialize where in-process, isolate where not"; (5) cwd/env process-globals → §4 explicit threading + audit; (6) elicitation is connection-local → §3.5 popup stays in the stdio front-end; (7) 10ms warm vs 1.3–7s cold → §7 hooks-first ladder.

## 1. TRANSPORT + DISCOVERY

**Transport: `multiprocessing.connection` — AF_PIPE named pipe on Windows, AF_UNIX socket on POSIX.** Stdlib, zero new dependencies (the library-knowledge boundary test passes trivially: it's the standard library), one code path across both OSes via `Listener`/`Client`, with the built-in HMAC `authkey` challenge as the authentication gate. Localhost TCP is rejected: it needs port allocation + a discovery race anyway, is reachable by any local user absent auth, and buys nothing a pipe doesn't. Raw pywin32 named pipes are rejected: a new dependency to re-implement what stdlib owns. **Framing over the connection is `send_bytes`/`recv_bytes` of UTF-8 JSON — never pickle** (a `recv()` unpickle of client bytes is an RCE-shaped hole; the daemon must never unpickle).

**Address**: `\\.\pipe\hpc-agent-<user>-<fingerprint12>` on Windows; `<daemon_dir>/sock-<fingerprint12>` (dir mode 0700) on POSIX — per memo B3-technical, the user AND the full build fingerprint (`_build_info.full_version()`, `src/hpc_agent/_build_info.py:138`) are IN the address. This makes cross-version connection structurally impossible through discovery: a stale pre-fix daemon from a dead session (incident 2) is simply at a different address than any post-reinstall client will ever dial.

**Discovery file**: `~/.hpc-agent/daemon/daemon-<fingerprint12>.json` (env-overridable `HPC_DAEMON_DIR` — tests and multi-install isolation need it), written by the daemon at ready via `atomic_write_json` (`src/hpc_agent/infra/io.py:110` — the one crash-durable JSON writer). Contents: `{address, pid, authkey_hex, full_version, fingerprint, protocol_rev, started_at, python}`. **User-only ACL**: on POSIX `chmod 0600` file + 0700 dir; on Windows the file lives under `%USERPROFILE%` (inherited user DACL) and the authkey is the real gate on both platforms — possession of the discovery file's secret is the credential, so the file's ACL is the ACL of the daemon.

**Staleness detection** (client side, in order, all bounded): (a) file absent → no daemon; (b) `pid_alive(pid)` false (`hpc_agent.infra.proc.pid_alive` — the single PID-liveness definition, re-exported at `detached.py:53`) → stale: delete file, proceed as absent; (c) connect + authkey challenge + `x-hpc/hello` handshake (§2) fails or version-mismatches → treat as absent and fall inline; never block on a half-dead daemon (hard client deadline ≈ 250ms connect+hello). A stale file is only ever deleted after the dead-pid check — never on a mere connect failure (the daemon may be mid-boot).

**Single-instance lock**: the daemon holds a blocking `advisory_flock` on `<daemon_dir>/daemon-<fingerprint12>.lock` (`infra/io.py:377` — filelock: msvcrt byte-range on win32, fcntl on POSIX; the OS releases it on any death, so no stale-lock hazard) for its whole lifetime. A second instance attempts non-blocking acquire, fails, reads the discovery file, and exits 0 (memo B3: "second instance detects and exits"). Activation racers coordinate on a separate short-lived spawn lock (§6).

**Multi-session semantics: ONE daemon per (user, build-fingerprint) — two Claude sessions share it.** Justification: (a) the warm state is per-*install*, not per-session — the `@primitive` registry is read-only after registration (the state-leak audit in `mcp_server.py:556-585` already establishes this for the shared-process case), and the asyncssh pool's per-host slot cap is a fleet-wide cluster-social courtesy (`ssh_engine.py` invariant 2) that TWO daemons would double-book: two idle login-node sessions per host and two independent breakers racing the same throttle evidence; (b) all cross-call durable state is on disk per-experiment-dir under per-write locks (§5), so two sessions through one daemon interleave exactly as safely as two one-shot CLIs do today; (c) per-call context (§4) makes the daemon experiment-dir-agnostic, so "session" has no server-side meaning. Two *installs* (different fingerprints) get two daemons by construction — that is the desired behavior during a wheel swap, and the old one drains itself (§6).

## 2. PROTOCOL

**Decision: one protocol — the existing JSON-RPC 2.0 / MCP framing, served by the same transport-free `McpServer.handle()` core (`mcp_server.py:1567` takes a request dict, returns a response dict — it is already transport-agnostic by design), extended with a private `x-hpc/*` method namespace for daemon concerns.** A second protocol is rejected on one-definition grounds: `tools/call` already owns verb→argv marshalling (`_build_invocation`, `mcp_server.py:896`), spec-temp-file mechanics (`_invoke_cli`, :1231-1238), and the envelope→result mapping that preserves `error_code`/`category`/`retry_safe`/`exit_code` (`_tool_result`, :942) — re-implementing any of that in a "daemon RPC v1" is exactly the mirror-twin class `lint_mirror_ledger.py` exists to catch. Zero new deps: `json` + `multiprocessing.connection`.

Three method groups over the pipe:

1. **`x-hpc/hello`** (per-connection, mandatory first message): `{client_full_version, client_fingerprint, pid, surface: "hook"|"cli"|"mcp-shim"|"block-drive", transport_env: {…}}` → `{ok, server_full_version, fingerprint, dist_signature}`. Version handshake per memo B1: any fingerprint mismatch → typed refusal (`daemon_version_skew`), client falls inline byte-identical, **and the daemon initiates graceful drain + self-exit** (§6) — a mismatch reaching the handshake at all means discovery was bypassed or half-swapped, i.e. the daemon is provably stale. `transport_env` carries the client's `HEALABLE_TRANSPORT_ENV_VARS` values (`infra/env_flags.py:44`) for the §4 transport-policy check.
2. **`x-hpc/cli`** (the thin-client verb call): params `{argv, cwd, hpc_env: {all HPC_* verbatim}, deadline_ms}` → result `{exit_code, stdout, stderr}` — exactly the `CliRunner` contract (`mcp_server.py:327`), so the daemon-side implementation IS the in-process runner with per-call context (§4), not a new dispatch path. This is one added *method*, not a second protocol: it funnels into the same `cli.dispatch._invoke_parsed`-rooted core the MCP `tools/call` path drives, and the memo's per-RPC disciplines (argparse `SystemExit` caught per-RPC — already done at `mcp_server.py:616-617`) come along for free.
3. **`x-hpc/status` / `x-hpc/shutdown`** — observability + administrative drain (§6). MCP methods (`tools/*`, `resources/*`, `prompts/*`) remain answerable over the pipe for a future non-stdio MCP client, but the Claude MCP surface stays on the stdio front-end (§3.5).

**Wheel-identity check per-call**: before dispatching each RPC the daemon re-checks `installed_dist_signature()` (`src/hpc_agent/cli/_fast_path_cache.py:60` — the existing sorted-distribution-set hash; memo G6 prices it ~5ms) against the value captured at boot, **plus, for editable installs (dev tree), a stat of the package root's key files** (the dist signature can't see a `git pull` under an editable install — the same reason memo §0.10 forces dirty checkouts to always walk). On mismatch: finish the current call's reply (never tear an in-flight verb), then drain+self-restart (§6). This closes incident 2's "stale pre-fix server pairs surviving hours": the maximum staleness window is one call, not one session.

**What the reply timing means**: a reply is written only after the verb has returned, and for journal-appending verbs the verb returns only after the append seam's fsync (§5) — so "acked" always implies "durable" (memo D3, unchanged).

## 3. CONCURRENCY MODEL

The engineered answer to incidents (3) and (4), stated as a rule first: **serialize where dispatch is in-process; isolate in a subprocess where it is not; keep long-running work in durable detached workers that survive the daemon.** Three lanes plus a supervisor:

### 3.1 Lane A — the serial fast-verb queue
One dispatch worker thread consuming a bounded `queue.Queue` (depth 32; overflow → immediate typed refusal `daemon_busy`, `retry_safe: true`, client falls inline — the queue must never become an invisible wait). Admission is a **DECLARED, ENUMERATED eligibility list** — the daemon extends unit 2.3's in-process list, same planted-violation contract test (`tests/contracts/test_src_subprocess_timeout_discipline.py` pattern): read/query verbs, `append-decision`, `block-drive` ticks whose spans are themselves eligible, `poll-detached`, `read-decisions`, `verify-relay`, etc. — the ms-class verbs. Eligibility criteria (each mechanized): no ssh, no `os.chdir` (lint), no unbounded wait, declared ceiling ≤ `FAST_LANE_DEADLINE_SEC` (default 20s).

**Why serial solves the global-stream race by construction**: `_in_process_cli_runner` (`mcp_server.py:536-621`) captures via `contextlib.redirect_stdout`/`redirect_stderr` + `_shield_real_stdin` (:513-533) — all three rebind **process-wide** state. With exactly one in-process dispatch at a time, no two captures can ever overlap; the shield/redirect discipline that already survives the 17243a17 class (enforcement row: "In-process dispatch over mcp-serve never touches the real sys.stdin", `engineering-principles.md` MCP-runner map) is inherited unmodified. **The named choice is: serialize dispatch in v1; per-call worker subprocess for everything slower (Lane B). The thread-local stream-router alternative (installing a threadlocal-delegating proxy as `sys.stdout`/`stderr`/`stdin` once at boot, enabling K concurrent in-process dispatches) is explicitly deferred to a post-proving v2 flag** — it is a real design (pytest's capture uses it) but it changes stream identity for every verb and re-opens the heartbeat-thread bleed the pre-redirect-handle capture at `mcp_server.py:1255` was built to dodge; serial at ~10ms/call serves 3 hooks + a handful of reads per turn with zero contention worth fixing yet.

**Head-of-line protection — the deadline seam, three rungs**:
- *Admission*: nothing without a ≤20s declared ceiling enters Lane A (the same "refuse at the seam, never in prose" posture as `_refuse_blocking_over_mcp`, `mcp_server.py:701`, which the daemon keeps enforcing verbatim for `x-hpc/cli` too — a blocking watch without `detach:true` is refused with the identical message whether it arrives over stdio or the pipe).
- *In-daemon backstop*: SIGALRM is a win32 no-op (`_in_process_deadline`, `mcp_server.py:487-489`), so the daemon's backstop is a watchdog thread: when the lane's in-flight call exceeds its ceiling ×1.5, the daemon marks itself WEDGED — stops accepting connections, logs the offending argv, and self-exits when the call returns (or hard-exits after a 60s grace via `os._exit` after flushing the log; a wedged daemon holding no locks (§5) and owning no durable state (§3.3) is safe to shoot).
- *Client-side guarantee (the load-bearing one)*: every thin client sets `deadline_ms` and an equal local timer; expiry → abandon the pipe call and **run inline** (fallback doctrine). A wedged daemon therefore costs each surface one bounded wait, never a wedge — the exact inversion of tonight's incident 3, where the server's wedge WAS the surface's wedge.

### 3.2 Lane B — per-call subprocess for bounded-slow verbs
Everything CLI-dispatchable but not Lane-A-eligible (ssh-shelling preflights, `resolve-resources`, `net-triage`, unit 2.3's KEEP-subprocess list including the census waiter) dispatches as `python -m hpc_agent <argv>` through the existing bounded tree-kill runner (`infra.bounded_subprocess.run_capture_bounded` / `capture_via_select` — the run-#7 enforcement rows: POSIX killpg / Windows `taskkill /T`, so an orphaned grandchild ssh can never outlive the deadline holding the pipe). Concurrency here is process-granular and unlimited-ish (a small semaphore, default 4): processes give cwd/env/stream isolation for free, which is why Lane B needs none of Lane A's discipline. Latency parity with today's cold CLI is accepted: these verbs' wall-clock is SSH-dominated; the daemon's 10ms-vs-1.3–7s win (incident 7) was never theirs.

### 3.3 The task supervisor — long-running work
Long-running verbs (the `SUPPORTED_DETACHED_BLOCK_VERBS` set — `detached.py:104-115`: S2/S3/S4, speculative canary, status-watch, aggregate-run/flow, campaign-run) **keep today's detached-worker machinery unchanged as the durability substrate**. The decision on ownership: **the on-disk state stays authoritative — the `(run_id, block)` lease with its holder pid (`detached.py:65-71`, self-healing on dead pids per `DetachedLeaseHeld`, :122-133), the journal stamps the workers write as they poll, and the block-terminal records. The supervisor owns only: launch dedup (it calls the same `launch_submit_block_detached`), an in-memory watch table for `daemon-status`, and wake fan-out (notifying a waiting client when a lease's terminal lands).** A daemon crash therefore loses exactly an in-memory table that is fully reconstructable by scanning `_detached/` leases — leases and terminals never lived in the daemon. This is deliberately NOT "move the watches into daemon asyncio tasks": coupling an hours-long S3 watch to a process that self-restarts on every wheel swap (§6) points the lifetime dependency the wrong way. Detached workers keep their own per-process ssh engines; re-opening in-daemon watches requires a ruling, recorded as declined-for-now.

### 3.4 The asyncssh pool
The daemon inherits `mcp-serve`'s role as *the* long-lived process where the persistent engine amortizes (`cli/mcp.py:21-35` — the invariants were hardened FROM mcp-serve incidents). One pool per daemon: one loop thread (`ssh_engine.py:309-334`), per-host connections, breaker-gated opens, slot-held-while-open, `IDLE_CLOSE_SEC=120` sweeper (:142), `atexit` shutdown — all unchanged. Lane-A verbs that do a bounded cluster read get the warm channel; Lane-B subprocesses and detached workers dial their own (unchanged). One-daemon-per-user (§1) is what makes the pool a fleet-wide good citizen rather than a multiplier.

### 3.5 Elicitation stays in the stdio front-end — the daemon never elicits
Incident 6, engineered: the popup machinery (`_elicit_then_retry` `mcp_server.py:1312`, the bidirectional pump `_request_from_client` :1672, per-session dark-channel state :1069) is **connection-local by nature** and stays in the `mcp-serve` stdio process. When mcp-serve becomes a thin shim (§7 rung 3), the flow for `append-decision` is: shim forwards to daemon → daemon returns the authorship refusal envelope (with the `authorship_evidence` key in `failure_features`, :356-357) → the SHIM's `_elicitation_applies` fires exactly as today → popup → `append_utterance` (a locked file append, safe from any process) → shim re-forwards the identical call once. Consequences: (a) a CLI/hook-originated call that needs a popup gets the refusal envelope byte-identical to today — the pipe surface never declares elicitation, so the gate at `mcp_server.py:1300` structurally cannot fire; (b) a 300s human wait NEVER occupies the daemon's serial lane — the wait lives in the one process that owns that one client's connection. This also keeps the daemon's per-connection state trivial (no popup state machine × N clients).

## 4. PER-CALL CONTEXT

**experiment_dir — threaded explicitly, never chdir, resolved at the CLIENT.** The contract: the thin client ALWAYS materializes `--experiment-dir <abs>` into the argv it sends, defaulting to its own cwd when the caller didn't pass one (every subcommand accepts it — `cli/dispatch.py:21`). The daemon never calls `os.chdir` and never trusts its own cwd (it starts in `~`). The request's `cwd` field is carried for telemetry only. This converts the process-global into an argv fact before it ever crosses the pipe.

**env — the split contract, decided:**
- **Transport-selection vars are DAEMON-side policy.** `HEALABLE_TRANSPORT_ENV_VARS` (`env_flags.py:44-52`: `HPC_SSH_ENGINE`, `HPC_SSH_CIRCUIT_OVERRIDE`, `HPC_NO_SSH_MULTIPLEX`, `HPC_SSH_BINARY`, `HPC_CLUSTERS_CONFIG`) configure the pool the daemon *owns*; they are snapshotted at daemon boot, echoed in the hello reply and `daemon-status`, and — the enforcement — a client whose own transport values disagree with the daemon snapshot is refused (`daemon_env_mismatch`, `retry_safe: true`) and falls inline, where its env genuinely applies. Silently serving a client that exported `HPC_SSH_ENGINE=native` from a pool running asyncssh would be exactly the env-vs-record drift class `active_env_overrides` (:63-78) exists to disclose.
- **Everything else `HPC_*` is CALL-side.** The client snapshots all its `HPC_*` vars at call time into `hpc_env`; the daemon threads them per-call through **one new seam: `env_flags.call_env()` — a `ContextVar[Mapping[str,str]]` defaulting to `os.environ`**, which `env_flag`, `env_actor`, and `active_env_overrides` are refactored to read through. Lane A's dispatch thread sets it for the call's duration (contextvar, not `os.environ` — the ssh-engine loop thread, sweeper, and heartbeat threads keep seeing the daemon's own env, which is correct because everything they read is daemon-side policy by the split above). Lane B applies `hpc_env` to the child's `env=` — full fidelity for free. This satisfies memo row L438-441's posture and its enforcement row verbatim: `HPC_ACTOR` still has exactly one reader (`env_actor`, `env_flags.py:93`), now resolving from the CALL's env — unit 3.3's "actor from call env" acceptance test.
- Calls whose critical env cannot be applied are refused, per memo B2 ("refuse calls whose critical env cannot be applied") — refusal, never approximation.

**cwd/env-independence audit — the grep patterns build units MUST run, with today's hits pre-classified:**
1. `Path\.cwd\(\)|os\.getcwd\(\)|os\.chdir\(` over `src/hpc_agent` — 26 hits today. Classification buckets: *client-side argparse defaults* (`cli/_helpers.py:255`, `block_drive.py:1384`, `incorporation/scaffold_strategy.py:230` — fine, resolved before the pipe); *hook payload fallbacks* (`_kernel/hooks/*.py` — hooks read the harness payload's `cwd` field first, e.g. `relay_audit_stop/__init__.py:221`, `utterance_capture.py:96`; the autofetch pair's bare `os.getcwd()` at `decision_rendezvous_autofetch.py:125` / `skill_return_autofetch.py:228` is fine because hooks run in their own process — but must be flagged if a hook ever becomes a daemon-side dispatch); *detached-worker contract* (`cli/dispatch.py:290` in `_record_detached_failure_terminal` — worker cwd is set by `_spawn_detached`, subprocess, fine); ***daemon-reachable verb bodies — the fix list***: `state/journal.py:169` and `state/index.py:275,325` (`Path.cwd()` when `experiment_dir is None` — must never be reached with None from daemon dispatch; add an assert-on-daemon guard), `cli/setup.py:103,232`, `infra/backends/__init__.py:551,680`, `experiment_kit/_runtime.py:284`, `ops/monitor/watcher_install.py:394`, and the one genuine `os.chdir` at `ops/notebook/scaffold_template_op.py:110` — that verb is Lane-B-only until fixed (the eligibility lint makes this mechanical, not remembered).
2. `os\.environ` (read) outside `env_flags.py` / `_fast_path_cache.py` / `ssh_engine.py` / `ssh_options.py` / detached-spawn sites — every `HPC_*` read must route through the `call_env()` seam; extend the existing sole-reader contract test pattern (`tests/contracts/test_multi_human_boundary.py::test_env_actor_is_the_sole_reader_of_hpc_actor`) into a general `HPC_*`-reader census with an allowlist.
3. `sys\.std(out|err)\.write|print\(` inside Lane-A-eligible verb bodies for progress lines — the memo D2 rule: heartbeats/progress take a stream parameter, never resolve `sys.stderr` at write time (the `mcp_server.py:1250-1264` lesson generalized).
4. Fire test (per eligible verb): call it twice through one daemon from two fake cwds + two `hpc_env` overlays → different journals touched, different actor stamped, byte-identical envelopes vs inline runs.

## 5. JOURNAL AUTHORITY

**Mechanics (memo D3, made concrete):** every daemon-dispatched journal append routes through the one seam `infra/io.append_jsonl_line` (`io.py:257-287`) — which already does, inside one call: parent mkdir → `advisory_flock` on `<path>.lock` → append → `flush()` → `fsync(fd)` → release. The daemon adds no second write path (one-definition; the append-seam lint from the memo enforcement rows covers it). Reply ordering is structural: the RPC reply is serialized only after the verb handler returns, and the handler returns only after the seam's fsync — no new machinery, one new fire test (row 17): dispatch `append-decision` via the daemon, `TerminateProcess` the daemon at the instant the reply is received, re-read the journal: record present, parseable, complete.

**Two hardening deltas the build unit must decide loudly, not silently:**
1. `append_jsonl_line` currently wraps the fsync in `contextlib.suppress(OSError)` (`io.py:286-287`). Under D3 "reply after the seam's fsync *returns*", a suppressed fsync failure means an ack without durability. Proposal: split the seam's callers — source-of-truth ledgers (decision journal, utterances, block terminals) get `fsync_required=True` where an fsync `OSError` raises (surfacing in the envelope as a non-ok, no reply-as-success); best-effort markers (harvest guard) keep suppression. Fire test: fsync monkeypatched to raise → `append-decision` returns ok:false, no ack.
2. First-append-to-a-new-file durability: appending fsyncs the file but not the parent dir, so a brand-new journal's dirent can be lost to a crash even though the data blocks were flushed. Extend the seam: when the open created the file, `_fsync_dir(parent)` (`io.py:92`) after the write — same best-effort NFS posture as the atomic writers.

**Lock interop with fallback one-shot writers:** the exclusion IS the per-write flock inside the seam — a one-shot CLI writer and the daemon contend on the identical `<path>.lock` via the identical filelock backend (msvcrt/fcntl, `io.py:377-446`), so mixed-mode operation (some surfaces on the daemon, some fallen inline) serializes exactly as two CLIs do today. **The daemon must never hold a lock across calls — and structurally cannot**, because no daemon-level code acquires locks; only verb bodies do, and every acquisition in the append path is a `with` block inside one call. Pinned by a contract test: after each daemon RPC completes, every journal/lease `.lock` in the test experiment dir is acquirable non-blocking from the test process. The 120s lock timeout (`io.py:283`) already converts a wedged holder into a loud, path-naming `TimeoutError` (run-#12 finding 16) — that surfaces in the envelope like any verb error.

**Crash semantics — what an acked write guarantees:** (a) the record's bytes are on stable storage and whole-line-atomic against concurrent writers; (b) it will be visible to every future reader including one-shot fallback CLIs (same file, same lock). What a MISSING ack guarantees: nothing except never-torn-prior-records (append-only + flock). A client that sent a mutating call and got no reply (daemon died post-fsync/pre-reply, or client deadline expired) must treat the outcome as **UNKNOWN and re-drive through the verb** — the verbs' existing idempotency (recorded-terminal replay, decision dedup) resolves it — and must never blind-append a raw line to "fix" it. One fire test per side: kill pre-fsync → no record, retry cleanly re-appends; kill post-fsync/pre-reply → record present, retry is a no-op/replay, no duplicate semantic decision.

## 6. LIFECYCLE

**Activation (client starts daemon if absent):** connect attempt (~1–5ms) → on absent/stale: non-blocking flock on `<daemon_dir>/spawn.lock`; the winner spawns **`python -m hpc_agent daemon-serve`** (memo B1: never the console-script `.exe` — the WinError-32 reinstall-lock class) with the same platform detach flags `_spawn_detached` uses, then waits bounded (≤15s; WS-BAKE's fast boot is a declared dependency of acceptable spawn latency) for the discovery file; losers poll the same file. Any failure/timeout → **run inline now** — activation can never cost a surface more than one bounded wait. **Hooks use fire-and-forget activation**: no daemon → run inline immediately AND leave the spawn running in the background, so the turn's remaining hooks (and every later call) are warm; a hook never pays the boot.

**Idle-exit:** after `HPC_DAEMON_IDLE_EXIT_SEC` (default 2h) with zero open connections, zero in-flight RPCs, and zero Lane-B children → `ssh_engine.shutdown_all()`, delete discovery file, release instance lock, exit 0. **Detached workers are NOT waited on and MUST NOT be**: their durability contract is precisely that they outlive every parent (`detached.py:1-35`); their leases/terminals ARE the durable task handoff, permanently, not an exit-time action. "Idle-exit AFTER last supervised task" therefore means: after the last *daemon-owned* unit of work (in-flight RPC / Lane-B child) — the disk-durable tasks were never the daemon's to lose.

**Stale-instance cleanup:** at boot, scan the daemon dir: discovery files with dead pids → delete; live pid + *different* fingerprint → send `x-hpc/shutdown` (graceful; it will also self-detect via §2's per-RPC check on its next call regardless). `doctor` gains the memo-mandated check: enumerate discovery files, pid liveness, fingerprint vs `installed_dist_signature()`, and flag `fingerprint-skewed daemon` with the stop command named.

**Wheel-swap self-restart:** per-RPC identity check (§2) trips → finish the in-flight reply → DRAINING: refuse new connections and new RPCs with `daemon_stale` (clients fall inline, and their next activation spawns the new-fingerprint daemon at its new address — no collision, no coordination) → close listener → exit when in-flight Lane-B children finish or after a 30s hard deadline. The `release` skill gains a `hpc-agent daemon-stop` step before install (running `python.exe` holds package file handles on Windows; the per-RPC check is the backstop, the stop step is the courtesy). `install_agent_assets`/program-init adds the same stop to the reinstall sequence.

**Logout/reboot:** the pipe and process die with the session; the discovery file goes dead-pid-stale; the next client cleans and respawns. No service, no autostart — on-demand only, by design (a daemon nobody is calling should not exist).

**Observability:** new curated read verb **`daemon-status`** (and its `x-hpc/status` twin): `{running, pid, address, full_version, fingerprint, uptime_s, calls_served, fallback_refusals{busy, env_mismatch, stale}, lane_a{depth, in_flight_verb, in_flight_ms}, lane_b_children, detached_tasks (from the _detached/ scan), ssh_pool[{host, alive, idle_for_s, inflight}], connections[{surface, pid, age_s}], last_identity_check_ms}`. Plus a size-capped rolling log `<daemon_dir>/daemon-<fingerprint12>.log` carrying the `[mcp]`-style per-call telemetry lines (`mcp_server.py:1272`) and lifecycle events — "why is it slow is a measurement, not a mystery" now holds for the daemon too. Daemon-down `daemon-status` reports honestly from the discovery file + pid check (never "no daemon" when a stale file names a live pid it can't reach — that IS the finding).

## 7. SURFACE MIGRATION LADDER

## 8. TEST STRATEGY

**CI shape:** the daemon battery runs on BOTH matrix legs, with the **Windows leg mandatory and gating** (transport is Windows-first; the false-EOF class only exists there — same reasoning as the existing `test_serve_survives_sequential_tool_calls_over_real_pipes` @slow row). All daemon tests take `HPC_DAEMON_DIR=<tmp>` so no test ever touches a user's real daemon; the spawned daemon is a REAL `python -m hpc_agent daemon-serve` subprocess, never a mocked listener.

1. **Live-daemon concurrency test:** spawn one daemon; fire N=8 concurrent clients (mix of threads and subprocesses, mimicking hooks+CLI+shim) each running a sequence of mixed reads + `append-decision` into DISTINCT tmp experiment dirs with distinct env overlays; assert per-call envelope parity against inline runs (parity oracle = `_subprocess_cli_runner`, the established pattern in `tests/test_mcp_curated.py`), correct journal routing per §4's fire test, and zero cross-call bleed in stdout captures (a canary marker printed by call k never appears in call k±1's envelope — the incident-4 fire path).
2. **Soak-sequential extension of the 17243a17 regression:** 200 sequential calls over the real pipe through one daemon, interleaving (a) a verb that attempts a stdin read, (b) a verb whose dispatch runs the UTF-8 `reconfigure` loop (`dispatch.py:383-387`), (c) ordinary reads; daemon alive and answering at call 200. This is the same invariant as `tests/test_mcp_server.py::test_serve_survives_sequential_tool_calls_over_real_pipes` lifted onto the new transport — the second-call-dead-server class must have a fire path on every transport it can occur on.
3. **Fire-path tests per invariant** (each named to its enforcement row):
 - *Durability (row 17):* kill daemon at reply receipt → record durable; kill pre-fsync (fault-injection seam in the append) → no record + clean retry; fsync-raises → no ack (the §5 hardening).
 - *Per-call context (row 18):* two calls, two cwds/envs → two journals, actor from call env; planted `os.environ["HPC_..."]` read outside the seam turns the reader-census contract test red; planted `os.chdir` verb in the Lane-A list turns the eligibility lint red (the 2.3 planted-violation pattern).
 - *Skew:* rewrite the dist-info sentinel mid-session → next RPC refused `daemon_stale`, client output byte-identical to inline, daemon exits ≤30s; hello with a wrong fingerprint → refusal + self-exit; stale discovery file (dead pid) → cleaned + respawned.
 - *Single instance:* two racing activations → exactly one daemon, both clients served (win32 spawn-lock race).
 - *Head-of-line:* planted slow verb in Lane A (test seam) → concurrent client falls inline within its deadline; watchdog marks WEDGED and the daemon self-terminates; `_refuse_blocking_over_mcp` refusals verified over the pipe surface too.
 - *Lock discipline:* post-RPC, every `.lock` acquirable non-blocking from the test process.
 - *Elicitation locality:* authorship-refused `append-decision` over the pipe → envelope identical to today's CLI refusal, no popup attempt; through the rung-3 shim with declared elicitation → popup fires in the shim, retry-once bound holds (existing E4 tests re-pointed).
 - *Lifecycle:* shrunken idle timeout → exit only after in-flight work, discovery file removed, engine `shutdown_all` observed; a detached worker launched via the daemon keeps running across daemon kill+respawn and `poll-detached` still reads it (the §3.3 crash-loses-nothing claim, exercised).
4. **Named batteries** (per unit 3.3, extended): `tests/integration/test_dispatch_smoke.py` gains the daemon leg; `tests/test_mcp_curated.py`; `tests/test_mcp_server.py -m slow`; the `tests/cli/` fast-path trio (daemon path must not regress the fast path's own pins); conformance lane both adapters; new `tests/daemon/` package for the above. Lints: the eligibility lint, the `HPC_*`-reader census, `lint_atomic_durable_writes.py` on the discovery file, `lint_subject_imports`/`lint_private_cross_package_imports` on the new modules. Enforcement-map rows 17/18 plus new rows for: fingerprint-in-address, transport-env split, client-deadline-falls-inline — appended to `engineering-principles.md` at wave integration per the standing protocol.

**New files:** `src/hpc_agent/cli/daemon_client.py` (stdlib-only thin client — import weight IS the product for rung 1), `src/hpc_agent/_kernel/extension/daemon_server.py` (listener, lanes, supervisor, identity checks — wraps the existing `McpServer` core), `src/hpc_agent/cli/daemon.py` (`daemon-serve`/`daemon-stop`/`daemon-status` registration). `mcp_server.py` changes are confined to: per-connection (vs per-instance) capability state, and exposing `_invoke_cli`'s core for the shim — the elicitation, refusal-seam, and projection code moves nowhere.

**Residual risks, named:** (1) `multiprocessing.connection` AF_PIPE under high client churn is the least-exercised stdlib corner in this design — rung-1 telemetry (fallback rate) is the tripwire, and the client deadline bounds the blast radius to "runs like today"; (2) the transport-env split refuses rather than adapts — a user flipping `HPC_SSH_ENGINE` mid-session sees inline fallbacks until the daemon idles out or is stopped (disclosed in the refusal message); (3) ContextVar env threading depends on the reader-census lint actually catching strays — the census allowlist must start empty-by-default and grow only with justification comments, or it rots into the thing it replaced.
