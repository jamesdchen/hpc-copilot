## Item 1 ‚Äî S1 reducibility shift-left

**Lead (as remembered):** reducibility problems surface only at S4/aggregate time;
shift the check left to S1 so an irreducible plan is refused before compute is spent.

**Ground truth.**
- The reducibility predicate is **`per_task_fallback_reducible(summary_name)`**
  (`src/hpc_agent/ops/aggregate_flow.py:305`, exported `:98`) ‚Äî TRUE iff the declared
  summary artifact is a JSON name the built-in no-combiner per-task fallback
  (`reduce_metrics`, a JSON weighted-mean) can actually reduce.
- It is **already lifted into a check-time gate**: `_reducibility_issue`
  (`src/hpc_agent/ops/aggregate_blocks.py:210`) reads the sidecar, and when a run
  has **no `aggregate_defaults.aggregate_cmd`** AND a **non-JSON summary artifact**,
  emits a never-auto-masked `non_reducible_summary_artifact` readiness issue. The
  docstring is explicit that "the condition is knowable HERE from the sidecar the
  submit wrote" (`:217-220`).
- **But that gate fires only at aggregate CHECK time** ‚Äî `aggregate_blocks.py:436`
  appends it to the aggregate readiness issues, i.e. **after the whole run has
  computed and after a 40-min `results/` pull** (`:216-217` names exactly this
  cost). This is finding-28's landing (`9f2760b5`, "surface ‚Ä¶ at CHECK time, before
  greenlight") ‚Äî a shift-left *within* aggregate, not to S1.
- **S1 does NOT run the check.** `submit_blocks.py`'s S1 leg discloses a
  `deploy_payload` brief (`:736`, finding-4 payload disclosure) and mints the
  sidecar via `resolve_submit_inputs` (`:719`), but there is **no reducibility
  call** anywhere in `submit_blocks.py` or `walk_submit_ambiguities.py` (grep:
  zero hits for `reducib` / `per_task_fallback` in either). The sidecar the check
  needs (`aggregate_defaults.aggregate_cmd` + `resolved_summary_artifact`) **exists
  by S1's resolved boundary** ‚Äî `rr.sidecar_path` / `rr.submit_spec` at
  `submit_blocks.py:719-728`.

**Verdict: OPEN.** The predicate is shared and cheap; only the call site at S1 is
missing. This is the smallest of the open items ‚Äî it reuses `_reducibility_issue`
verbatim against the just-written sidecar.

**Remaining spec (build-ready).**
- **Files touched:**
  - `src/hpc_agent/ops/aggregate_blocks.py` ‚Äî export `_reducibility_issue` (or lift
    it + `per_task_fallback_reducible` into a shared `ops/aggregate/_reducibility.py`
    so submit does not import an aggregate-block internal; layering-lint will demand
    the shared home ‚Äî check `scripts/lint_subject_imports.py`).
  - `src/hpc_agent/ops/submit_blocks.py` ‚Äî in the S1 resolved leg, right beside the
    `deploy_payload` disclosure (`:736`), call the shared predicate against
    `rr.sidecar_path` and attach a `reducibility` block to `brief` when it fires.
  - `tests/ops/test_submit_blocks.py` (or the S1 block test home).
- **The change:** at the S1 resolve boundary, run the same predicate the aggregate
  check runs; surface a `non_reducible_summary_artifact` **disclosure** on S1's
  brief (the human boundary before submit-s2 detaches and spends compute). Because
  the predicate is code-computed from the sidecar and fail-open, it mirrors the
  `deploy_payload` finding-4 pattern exactly.
- **USER RULING owed (one, small):** *disclose vs hard-refuse* at S1. Finding-4's
  payload is disclosed-not-blocking; a non-reducible plan is arguably worse (it
  **cannot ever** produce a number, so any compute is wasted). Recommend: surface as
  a **loud readiness disclosure that does not auto-mask** (same class as the check
  gate) and let the greenlight cross it deliberately ‚Äî NOT a hard block, since a
  human may intend to register the reducer between S1 and aggregate. The ruling is
  only "disclose vs block"; everything else is pure engineering.
- **Pins/tests owed:** fires-and-passes pair ‚Äî (a) an S1 spec with no `aggregate_cmd`
  + a `.csv` summary artifact ‚Üí S1 brief carries the reducibility disclosure; (b) a
  spec with an `aggregate_cmd` (custom reducer) OR a `.json` artifact ‚Üí clean S1,
  no disclosure. One test asserting the S1 predicate and the CHECK predicate are the
  **same call** (they can never disagree ‚Äî the finding-28 "lifted so the two can
  never disagree" invariant, extended to a third seat).
- **Enforcement row owed** (`docs/internals/principles/lifecycle-verdicts.md`,
  format `| Rule | Enforced by | Fires when |`):
  `| Reducibility is one predicate, now THREE seats (run-path refuse, aggregate-CHECK disclosure, S1 disclosure) ‚Äî all call per_task_fallback_reducible; no seat re-derives it | tests/ops/test_submit_blocks.py::test_s1_surfaces_non_reducible, tests/ops/test_aggregate_blocks.py::test_check_surfaces_non_reducible | an S1/check seat inlines its own JSON-name test instead of the shared predicate, or the S1 seat stops firing before submit-s2 detaches |`
- **Size:** SMALL (~1 shared-home extraction + 1 call site + 2 tests + 1 row).

## Item 2 ‚Äî write-run-sidecar not hand-edit

**Lead (as remembered):** S4/recovery/retarget flows have users/agents HAND-EDITING
run sidecars; the `write-run-sidecar` primitive exists ‚Äî the gap is flows that don't

**Ground truth.**

  the inline `python -c "import hpc_agent.state.runs; write_run_sidecar(...)"` agents
  used to make (primitive doc `:19`).
- **Every code writer routes through the primitive / the state-layer functions.**
  `write_run_sidecar` / `update_run_sidecar` importers (grep):

mechanized; all flows route through the primitives.

## Item 3 ‚Äî reducer-contract doc

**Lead (as remembered):** the contract between task outputs and the reducer lacks a
canonical doc; check whether the streaming Step-C reducer-contract fixes landed ‚Äî the
DOC may still be owed even if the fixes landed.

**Ground truth.**
- The canonical doc **EXISTS**: `docs/reference/reducer-contract.md` ‚Äî the four-point
  contract (reads `$HPC_RUN_ID`, writes `$HPC_AGGREGATED_OUTPUT`, valid JSON, exit
  code), cluster-vs-local (`$HPC_RESULTS_DIR`) semantics, a minimal example, and the
  scalar-mean‚Üícombiner / anything-else‚Üícluster-reduce split rule. It is referenced
  from `ops/aggregate/cluster_reduce.py:19,35`.
- **But it predates the streaming-era contract additions.** Last touched `d81ed2d0`
  (pure-API reduction honors mode/aggregate_cmd) ‚Äî BEFORE the streaming train
  `d2b18bfd`. It does **not** mention (grep for `stream` / `arm` / `activation` /
  `env_python` / `deploy` ‚Üí zero hits) the three things that landed since and are now
  part of the effective contract a reducer author must know:
  1. **Interpreter binding (S-REDUCE ¬ß3.C.1, `cluster_reduce.py:287-299`).** The
     reducer's literal `python3`/`python` now binds the **run's env interpreter** via
     `remote_activation_for_sidecar`, not bare login python (retires the run-14
     py3.8-vs-3.13 crash). A reducer author must know their command runs under the
     run's cluster env, and that a local-env miss is disclosed as the pinned
     interpreter, not a cluster bug.
  2. **Ship-the-reducer + submit-time refusal (S-REDUCE ¬ß3.C.2, item 4 below).** A
     **file-path** reducer (`python3 specs/reduce_x.py`) is auto-derived
     (`_deploy_items.py:69 reducer_relpath_from_aggregate_cmd`), shipped as a
     content-hashed deploy item, and **must be on disk at submit or the run REFUSES
     at stage** (`submit_flow.py:907-912`). A **module** reducer (`python -m pkg.x`)
     is assumed installed and is NOT shipped ‚Äî a materially different author contract
     the doc never states.
  3. **Streaming interaction (item 5).** `aggregate-stream` currently reduces via the
     **built-in** path only, so a custom-reducer run's streamed partial does not use
     the author's reducer ‚Äî the doc should warn that streaming ‚âÝ final for a custom
     reducer until item 5 lands.

**Verdict: PARTIAL ‚Äî the DOC is owed for the streaming-era additions** (the code
fixes landed; the canonical doc did not absorb them).

**Remaining spec (build-ready, pure docs).**
- **Files touched:** `docs/reference/reducer-contract.md` (extend, don't rewrite).
- **The change:** add two sections ‚Äî
  - *"Where the interpreter comes from"*: the reducer runs under the run's
    `env_python` activation; literal `python3` resolves to the run env; disclose the
    file-path-vs-`python -m` distinction and its shipping consequence.
  - *"Deployment & the on-disk requirement"*: a file-path reducer must exist under the
    experiment repo at submit; it ships automatically (content-hashed); a missing
    reducer refuses at submit, not mid-harvest. Cross-link
    `docs/primitives/aggregate-stream.md` with the "streaming uses the built-in path
    today" caveat (retire the caveat when item 5 lands).
- **Pins/tests owed:** none in the doc itself, BUT add a **doc-freshness pin** so the
  doc can't silently rot again: a contract test that greps `reducer-contract.md` for
  the `env_python`/`ship`/`stream` keywords, OR a row in the aggregate principles
  section naming the doc as the contract's source of truth. Recommend the latter.
- **Enforcement row owed** (`docs/internals/principles/determinism-boundary.md` or a
  new aggregate section):
  `| The reducer contract (interpreter binding, on-disk-ship refusal, JSON output) has ONE source-of-truth doc: docs/reference/reducer-contract.md; cluster_reduce + _deploy_items + submit_flow's stage-gate behavior match it | tests/reference/test_reducer_contract_doc.py (keyword pins) | the doc omits a shipped contract behavior, or a code seam diverges from the documented contract |`
- **Size:** SMALL (docs + one keyword-pin test + one row). **No user ruling.**

## Item 4 ‚Äî S4 pushes its own reducer

**Lead (as remembered):** S4/aggregate should deploy the reducer code it needs
(version-matched) rather than relying on whatever is on the cluster; check whether the

**Ground truth ‚Äî this LANDED in `d2b18bfd` (S-REDUCE ¬ß3.C.2).**
- **Derive:** `reducer_relpath_from_aggregate_cmd(aggregate_cmd)`
  (`src/hpc_agent/infra/transport/_deploy_items.py:69`) tokenizes the sidecar's
  `aggregate_defaults.aggregate_cmd` and extracts the `specs/reduce_*.py` file token
  (a `python -m` module reducer returns `None` ‚Äî it's assumed installed, `:81-86`).
- **Ship:** the derived reducer is added to the deploy set as a **content-hashed**
  item (`_deploy_items.py:221-227`) and forwarded through
  `submit_flow._resolve_reducer_deploy_item` (`submit_flow.py:858`) ‚Üí `deploy_runtime`
  as `extra_files` (`:853`), staged like `.hpc/_hpc_combiner.py`.
- **Stage-refuse:** a declared-but-absent reducer **refuses at submit**
  (`submit_flow.py:907-912`): "declares a custom reducer ‚Ä¶ NOT on disk ‚Ä¶ the cluster
  reduce would die mid-harvest with 'no such file' (the run-14 ‚Ä¶ manual-scp class)."
  The gate fires per-fresh-spec across a batch (`:884-891`).
- **Version-matched interpreter:** the shipped reducer runs under the run's env via
  the S-REDUCE ¬ß3.C.1 activation threading (`cluster_reduce.py:287-299`).

**Verdict: DONE.** S4 no longer relies on whatever reducer is on the cluster ‚Äî it
derives, ships (content-hashed), refuses-at-submit-if-absent, and runs it under the

**Residual (belongs to item 5, not this item):** the *streaming* path
(`aggregate-stream`) does not yet **invoke** the shipped custom reducer ‚Äî it reduces
built-in. The deploy/refuse machinery is done; the streaming consumer is not. See

## Item 5 (completeness sweep) ‚Äî aggregate-stream ignores the run's custom reducer

**Where it came from.** Sweeping the streaming spec's own ¬ß3.B / ¬ß6-S4 against what
`stream.py` actually shipped, plus the run-14 motivating artifacts
(`docs/plans/run14-rigor-runsheet.md:21-32` ‚Äî the live lgbm/xgb runs use
`aggregate_cmd = python3 specs/reduce_causal_tune_tree.py` emitting `qlike`/`dm_better`).

**Ground truth.**
- `aggregate-stream` (`src/hpc_agent/ops/aggregate/stream.py`) reduces complete arms
  through the **built-in** `reduce_metrics` weighted-mean (`:235,242`) or the
  ownership `multi_parent_reduce` (`:438`). It **never reads `aggregate_cmd`** and
  **never calls `cluster_reduce`** (grep: zero `aggregate_cmd` / `cluster_reduce`
  hits in `stream.py`).
- Consequence: for a run that declares a custom reducer ‚Äî **the exact live case the
  whole streaming train was built for** ‚Äî the streamed partial table carries built-in
  weighted-mean numbers that **do not match** the run's own reducer output
  (`qlike`/`dm_better` are not a plain mean of per-task scalars). And it does so
  **silently**: there is no refuse, no disclosure that "these numbers are not your
  reducer's."
- The streaming spec named this exactly (¬ß3.B, ¬ß5, ¬ß6 row S4): the intended contract
  was either an **`HPC_STREAM_ARMS` allowlist** the custom reducer honors, or a
  **per-complete-arm `cluster_reduce` + row-union** (reducer-agnostic). Grep confirms
  **neither shipped** ‚Äî no `HPC_STREAM_ARMS` in the tree. v1 shipped built-in-only.

**Verdict: OPEN.** This is the streaming train's honest residual and belongs in the
S4-gaps program because it is the same "every number is reducer-computed" doctrine
(streaming a built-in mean for a custom-reducer run violates it).

**Remaining spec (build-ready).**
- **Files touched:**
  - `src/hpc_agent/ops/aggregate/stream.py` ‚Äî add a reduce-path selector: if the run
    declares an `aggregate_cmd`, reduce each complete arm through `cluster_reduce`
    over that arm's dirs and union the rows (the spec's reducer-agnostic fallback);
    OR, minimally, **refuse-to-stream** with "run declares a custom reducer; streaming
    the built-in mean would not match ‚Äî final harvest only" (mirrors the existing
    non-wave-aligned refuse, `stream.py:386`).
  - `tests/ops/aggregate/test_stream.py`.
- **The change:** honor the custom reducer for streaming, or refuse honestly. Do NOT
  silently emit built-in numbers for a custom-reducer run.
- **USER RULING owed (one, real):** *invoke-per-arm vs refuse-to-stream* for the v1.
  Per-arm `cluster_reduce` is N SSH reduce invocations per census tick (cost) but
  gives a true progressive custom-reducer table; refuse-to-stream is cheap and honest
  but drops streaming for exactly the custom-reducer runs that motivated it. Recommend
  **refuse-to-stream in the immediate fix** (honest, ships now, zero wrong numbers)
  with **per-arm invoke as the follow-on** once the census-tick cost is acceptable.
  The ruling is behavioral (what the live lgbm/xgb operator gets), so it needs the
  user.
- **Pins/tests owed:** (a) a custom-`aggregate_cmd` run ‚Üí stream refuses (or invokes
  the custom reducer, per the ruling), never emits a built-in mean; (b) a
  built-in-reduce run ‚Üí unchanged. A doctrine test asserting stream never emits a
  number the run's declared reducer didn't compute.
- **Enforcement row owed** (`docs/internals/principles/determinism-boundary.md`):
  `| aggregate-stream emits only numbers the run's OWN reducer would compute ‚Äî a custom-aggregate_cmd run is reduced through cluster_reduce or refused, never through the built-in mean | tests/ops/aggregate/test_stream.py::test_custom_reducer_run_not_built_in_meaned | stream.py reduces a run with a declared aggregate_cmd through reduce_metrics |`
- **Size:** SMALL if refuse-to-stream; MEDIUM if per-arm invoke (touches the
  census‚Üídir‚Üíreduce plumbing + multi-invocation error handling).
