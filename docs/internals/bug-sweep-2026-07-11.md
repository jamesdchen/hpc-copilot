### 1. [HIGH] plan_block_action consumes a STALE greenlight: any latest y in the run scope — including the previous boundary's already-consumed approval — is treated as the approval of the CURRENT pending decision

- **Where:** `src/hpc_agent/_kernel/lifecycle/block_drive.py:233`  ·  **Category:** correctness  ·  **Confidence:** 2 skeptic votes
- **Failure scenario:** Same window as the stop-guard bug, driver side: run parked at the S2->S3 boundary (marker M2), journal's latest y is y#1 (which greenlit S1->S2 and was already consumed). _latest_committed_resolved returns y#1.resolved (it scans for the latest response=='y' with no check that the record targets this boundary, e.g. resolved.next_block == cursor.next_verb), so plan_block_action skips the `committed_resolved is None -> awaiting_decision` branch and routes by field-diff. run_tick then CLEARS the pending marker and executes: either it re-runs the current block (a wasted cluster-side replay per tick, re-parking after) or, when the diff is empty, it advances into the gated successor whose verb-level gate (assert_greenlit_target: latest y must NAME the verb) refuses with SpecInvalid — the span exits non-zero and the tick reports 'block submit-s3 failed or returned no result (exit 1)' instead of the true state 'awaiting the human'. - **Fix sketch:** Scope the resume-path approval to the current boundary: in run_tick, when a pending marker exists, accept the journal's latest y only if it targets the parked boundary — extract the greenlit verb from resolved["next_block"] (reuse block_gate._journaled_target semantics) and require it to equal the marker's resume_cursor["next_verb"] (and/or require the record's ts to be >= the marker's awaiting_since); otherwise pass committed_resolved=None to plan_block_action so the tick returns awaiting_decision/exit 0. Keep the unscoped latest-y scan only for the no-marker journal-derived resume (run #9). Apply the same boundary-matching predicate to find_committed_unadvanced in decision_rendezvous_stop_guard.py (and the doctor's twin check) so the stop guard stops forcing ticks during the stale window. Add a unit test: pending marker at current_verb=submit-s2/next_verb=submit-s3 plus a committed y whose resolved names submit-s2 must plan awaiting_decision, not advance.

### 5. [HIGH] PBS-family liveness/state queries permanently read UNKNOWN once any queried job leaves the queue, because scheduler_query_ran requires ack rc==0 but qstat exits non-zero for unknown/finished job ids

- **Where:** `src/hpc_agent/infra/backends/_engine.py:709`  ·  **Category:** correctness  ·  **Confidence:** 2 skeptic votes
- **Failure scenario:** On a pbspro/torque cluster, build_alive_check_cmd/build_scheduler_state_cmd query EXPLICIT ids: `qstat -t <id> ... 2>/dev/null; echo __HPC_SCHED_ACK__=$?`. Unlike SGE's `qstat -u $USER` (exits 0 on an empty queue — the rationale the rc==0 rule is built on), PBS qstat exits non-zero when a queried id is unknown/has finished ('Unknown Job Id' / 'Job has finished, use -x'). So the moment the first job of a run terminates, every poll returns ack rc!=0 → scheduler_query_ran returns ran_ok=False → reconcile._alive raises RemoteCommandFailed, cluster_status raises SshUnreachable, verify_submitted reads UNKNOWN. Consequences: a finished PBS run can never settle terminal from queue absence (rides the wait budget to unable_to_verify), kill confirmation (qdel then alive-check verification) can never confirm gone-ness, and mid-run with a mix of finished+running jobs the alive rows that WERE printed are discarded too. The SLURM branch special-cases exactly this failure mode for `squeue -j` (ack presence suffices); the PBS family has the identical explicit-id semantics but got the SGE rule. tests/infra/test_scheduler_states.py only exercises SGE and SLURM; no test covers PBS ran_ok.
- **Fix sketch:** In scheduler_query_ran (_engine.py:707-709), move the PBS family onto the SLURM rule: `if cls.profile.family in ("slurm", "pbspro", "torque"): return clean, True` — for explicit-id queries, ack presence proves the channel ran; a non-zero rc is the expected result of queried ids having left the queue, and the kept stdout rows remain parseable. (If distinguishing a truly failed qstat binary matters, additionally check for rc==127 or wrap the qstat invocation with `command -v qstat` — but ack presence already guarantees the shell executed the pipeline.) Update the docstring's per-family rules and the connection-broker.md table row accordingly, and add PBS cases to tests/infra/test_scheduler_states.py: (a) `PBS.scheduler_query_ran("123.svr R ...\n__HPC_SCHED_ACK__=153\n")` → ran_ok True with rows preserved, (b) ack absent → False, plus a build_scheduler_state_cmd shape test for pbspro/torque.

### 8. - **Where:** `src/hpc_agent/infra/ssh_engine.py:631`  ·  **Category:** concurrency  ·  **Confidence:** 2 skeptic votes
- **Failure scenario:** Enable the engine (HPC_SSH_ENGINE=asyncssh) and issue any capture ssh_run whose remote leg runs >120s — e.g. the 15-minute aggregate/combiner leg this module's own _LOOP_DEADLINE_MARGIN comment cites ("the live 2026-07-08 aggregate hang: a 15-min remote leg"), or any ssh_run with timeout=600/None. _HostConn.last_used is stamped only at connect (__init__) and AFTER a command completes (run(), line 518); nothing marks a connection busy while a command is in flight.
