# Domain packs: bind-as-data, trust content-addressed

A **domain pack** is the layer ABOVE core that NAMES what opaque content means
(which callables read data, what a failure pattern implies, which axis a
parameter is, what tolerance a metric deserves — `docs/design/domain-packs.md`).
Core never learns any of that meaning: it BINDS a caller-referenced pack's
files by raw-bytes sha (DP1), carries an opaque `{pack, version, sha}` echo on
every record that consumed pack content, and GATES on named pack RECEIPTS (code
attestations the pack's own execution emitted) — without ever running,
importing, or interpreting a line of pack logic (DP2/DP3). Trust is
CONTENT-ADDRESSED, not install-addressed: an edit to the bound standards moves
the sha and revokes every clearance signed under the old ones, the same
drift-revocation the notebook audit gives source code, extended to the domain
layer. This is the pack-specific restatement of the Q1 boundary above (IDENTITY
+ ORDERING + COMPARISON + COUNTING over opaque caller content); the moment a
core branch reads a declared VALUE for meaning, the line is crossed.
